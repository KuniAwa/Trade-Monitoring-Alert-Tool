# AI による分類精度向上と中区分の実現性検討

OpenAI Platform 利用を前提に、以下2点の実現性を整理する。

---

## 1. AI ツールで推定精度を上げること

### 結論：**実現可能**

### 根拠

| 観点 | 内容 |
|------|------|
| **技術** | OpenAI API（Chat Completions 等）に「ファイル名・親フォルダ名・拡張子」などを渡し、簿記の6分類（①資産～⑥その他）を返すプロンプトを書けばよい。短文の分類タスクは LLM の得意分野。 |
| **入力** | 現状の「名前」「親フォルダ」「拡張子」を 1 件ずつ（またはバッチで）API に送る。必要なら「前後のファイル名」などもコンテキストとして追加可能。 |
| **出力** | 応答を「①資産」～「⑥その他」のいずれかに固定（JSON や structured output で取り出す）。 |
| **コスト** | 1 ファイルあたり数十～百トークン程度。大量ファイルでも、現実的な予算で運用可能。無料枠・利用制限は要確認。 |
| **速度・制限** | 1 件ずつ呼ぶと時間がかかるため、複数件を 1 リクエストにまとめる、または並列化の検討が有効。Rate limit に注意。 |
| **オフライン** | API が使えない環境では、現行のキーワード推定にフォールバックする設計にすれば実現可能。 |

### 実装イメージ（概念）

- オプションで「AI 分類を使う／使わない」を切り替え。
- キーワードで一度推定 → その結果を AI に渡して補正する、または最初から AI のみで分類する、のどちらも可能。
- API キーは環境変数や設定ファイルで管理し、コードに直書きしない。

---

## 2. ①～⑥の大区分の中に「中区分」を設けること

### 結論：**実現可能**

### 根拠

| 観点 | 内容 |
|------|------|
| **定義** | 中区分は「大区分（①～⑥）の下のラベル」として定義すればよい。例：①資産 → 現金・預金／売掛金／棚卸資産／固定資産 など。 |
| **固定リスト** | 勘定科目や業界標準に合わせて、あらかじめ「大区分ごとの中区分リスト」を JSON や Python の定数で持つ。Excel の「分類」列の隣に「中区分」列を追加し、ドロップダウンで事後変更可能にできる。 |
| **AI で推定** | ファイル名などから「①資産」と「現金・預金」の両方を返すようプロンプトを設計すれば、AI で中区分まで推定できる。出力形式を「大区分＋中区分」の 2 段階にすればよい。 |
| **運用** | 中区分は追加・変更が発生しうるため、「中区分リストを外部ファイルで管理し、スクリプトが読み込む」形にすると拡張しやすい。 |

### 中区分の例（イメージ）

- **①資産** … 現金・預金、売掛金、棚卸資産、固定資産、その他資産  
- **②負債** … 買掛金、借入金、未払金、前受金、その他負債  
- **③資本** … 資本金、繰越利益剰余金、その他資本  
- **④収益** … 売上、営業外収益、その他収益  
- **⑤費用** … 売上原価、販管費、営業外費用、その他費用  
- **⑥その他** … （任意で「未分類」「その他」のみ、または細かく定義）

---

## 3. まとめ（実現性）

| 項目 | 可否 | 備考 |
|------|------|------|
| AI で分類精度を上げる | **可** | OpenAI API でファイル名等から大区分を推定可能。キーワードとの併用やフォールバックも可能。 |
| ①～⑥の大区分内で中区分を設ける | **可** | 大区分＋中区分の 2 段階で定義し、Excel では両方ともドロップダウンで事後変更可能にできる。 |
| AI で中区分まで推定する | **可** | プロンプトで「大区分＋中区分」を返すよう指定すればよい。中区分リストを API に渡して選択肢を限定することも可能。 |

実装する場合は、  
(1) API キー管理とエラー・オフライン時の扱い、  
(2) 中区分リストの管理方法（ファイル／スプレッドシート）、  
(3) コスト・レート制限を踏まえたバッチ or 並列の設計  
を決めてから進めるとよい。
